# 📌 결제 시스템 리팩토링 과제

## 🎯 과제 개요

기존의 결제 시스템 코드를 분석하고,
객체지향적으로 구조를 개선하는 것을 목표로 합니다.

---

## 1️⃣ 기존 코드의 문제점

Payment 클래스의 필드가 public으로 노출되어 있어 외부에서 함부로 금액이나 타입을 변경할 수 있어 데이터의 무결성이 훼손될 수 있습니다. 또한 결제 타입과 금액 데이터는 Payment에 있는 반면 이의 유효성을 검증하는 로직은 PaymentService에 있어 객체지향의 규칙에 위배됩니다. 마지막으로 PaymentService에 너무 많은 로직이 부담되고 있어 유지보수성이 떨어집니다.

👉 (힌트: 캡슐화, 책임 분리, Service 역할 등을 중심으로 작성)

---

## 2️⃣ 로직 이동 내용

### 🔹 변경 전

PaymentService 내부의 pay() 메서드에서 if (amount <= 0) 조건문을 통해 금액의 유효성을 검증했습니다.
PaymentService 내부에서 if (type.equals("card")) 와 같이 외부 데이터를 직접 꺼내와서 결제 로직을 처리했습니다.

### 🔹 변경 후

금액 유효성 검증 로직을 Payment 내부 생성재로 이동했으며 결제 처리 로직을 유효성 검증 로직인 isMatchedType(), 처리 로직인 executaPayment()로 나누어 이동하였습니다. 

👉 (어떤 로직을 어디로 옮겼는지 작성)

---

## 3️⃣ 구조 개선 내용

Payment의 필드를 private으로 감추고, 외부에서는 생성자와 공개된 메서드만을 통해서 소통하게 만들어 캡슐화 처리했습니다. PaymentService는 오직 결제 객체의 생성과 처리를 요청하는 역할에만 집중하게 되어 유지보수성을 향상시켰습니다.

👉 (예: 객체가 직접 일을 하도록 변경, Service 단순화 등)
